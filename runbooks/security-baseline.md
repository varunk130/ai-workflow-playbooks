# Security Baseline Runbook

Reference checklist for application security. Load this when performing security hardening or reviewing code for vulnerabilities.

## Pre-Commit Checks

Before any commit:
- [ ] No API keys, passwords, or tokens in source code
- [ ] No hardcoded credentials in configuration files
- [ ] `.env` files are in `.gitignore`
- [ ] No private keys or certificates committed
- [ ] Secrets scanner (e.g., git-secrets, gitleaks) passes

## Authentication

### Password Handling
- [ ] Passwords hashed with bcrypt, scrypt, or Argon2 (never MD5/SHA1)
- [ ] Salt is unique per user and generated by the hashing library
- [ ] Minimum password length enforced (8+ characters)
- [ ] Password reset tokens are single-use and time-limited (<1 hour)
- [ ] Failed login attempts are rate-limited

### Session Management
- [ ] Session tokens are cryptographically random (128+ bits of entropy)
- [ ] Cookies use `HttpOnly`, `Secure`, and `SameSite=Strict` (or `Lax`)
- [ ] Session expires after inactivity timeout
- [ ] Session is invalidated on logout (server-side)
- [ ] Session fixation protection in place (new session ID after login)

### Token-Based Auth (JWT)
- [ ] Tokens have expiration (`exp` claim)
- [ ] Token signature is verified on every request
- [ ] Refresh tokens are stored securely and rotated on use
- [ ] Algorithm is specified server-side (prevent `alg: none` attacks)

## Authorization

- [ ] Every endpoint checks authorization, not just authentication
- [ ] IDOR prevention: verify resource ownership before access
- [ ] Role/permission checks happen server-side, never client-only
- [ ] Admin endpoints require elevated privileges
- [ ] Rate limiting applied to sensitive operations

## Input Validation

### General Rules
- [ ] Validate on the server, even if client validates too
- [ ] Use allowlists over denylists (accept known-good, not reject known-bad)
- [ ] Enforce maximum lengths on all string inputs
- [ ] Validate numeric ranges (min, max, integer vs float)
- [ ] Validate email format and domain
- [ ] Sanitize file upload names and validate MIME types

### Injection Prevention
- [ ] SQL: Use parameterized queries / prepared statements (never string concatenation)
- [ ] HTML: Escape output or use a templating engine with auto-escaping
- [ ] Command: Avoid `exec` / `eval` with user input; use argument arrays
- [ ] Path: Validate and normalize file paths; prevent directory traversal (`../`)
- [ ] URL: Validate redirect URLs against an allowlist; prevent open redirects

## Security Headers

```
Content-Security-Policy: default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'
Strict-Transport-Security: max-age=31536000; includeSubDomains
X-Content-Type-Options: nosniff
X-Frame-Options: DENY
X-XSS-Protection: 0
Referrer-Policy: strict-origin-when-cross-origin
Permissions-Policy: camera=(), microphone=(), geolocation=()
```

## CORS Configuration

```javascript
// GOOD — explicit allowed origins
cors({
  origin: ['https://app.example.com'],
  methods: ['GET', 'POST', 'PUT', 'DELETE'],
  credentials: true,
});

// BAD — allows any origin
cors({ origin: '*' });

// BAD — reflects the request origin
cors({ origin: req.headers.origin });
```

## Data Protection

- [ ] Sensitive fields excluded from API responses (passwords, internal IDs, PII)
- [ ] Sensitive data not logged (passwords, tokens, credit card numbers)
- [ ] Data encrypted in transit (HTTPS everywhere)
- [ ] Data encrypted at rest for PII and financial data
- [ ] Backups are encrypted and access-controlled

## Dependency Security

- [ ] Run `npm audit` / `pip audit` / `cargo audit` regularly
- [ ] No dependencies with known critical CVEs
- [ ] Lock file committed and used in CI/CD
- [ ] License compatibility verified for all dependencies
- [ ] Unused dependencies removed

## OWASP Top 10 Quick Reference

| # | Vulnerability | Prevention |
|---|--------------|------------|
| 1 | Broken Access Control | Server-side authz checks on every request |
| 2 | Cryptographic Failures | Use standard libraries; no custom crypto |
| 3 | Injection | Parameterized queries; escape output |
| 4 | Insecure Design | Threat model during architecture phase |
| 5 | Security Misconfiguration | Secure defaults; no debug in production |
| 6 | Vulnerable Components | Audit dependencies; update regularly |
| 7 | Auth Failures | Strong hashing; MFA; rate limiting |
| 8 | Data Integrity Failures | Verify signatures; validate CI/CD pipeline |
| 9 | Logging Failures | Structured audit logs; monitor for anomalies |
| 10 | SSRF | Validate URLs; allowlist external destinations |
